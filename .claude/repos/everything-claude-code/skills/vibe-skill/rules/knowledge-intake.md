# 知识摄入规则

## 适用场景

- 用户提供领域知识、文档、最佳实践
- 用户说 "把这些转成 skill"、"基于这个创建 skill"
- 收到原始知识材料需要结构化

## 分析要点

| 步骤 | 分析要点 | 检索/操作 | 优先级 |
|-----|---------|----------|-------|
| 1 | 识别知识类型 | 按下表分类 | 必须 |
| 2 | 提取触发条件 | "何时应该激活这个 Skill?" | 必须 |
| 3 | 提取核心工作流 | "专家遵循什么步骤?" | 必须 |
| 4 | 识别决策点 | "哪里需要判断力?" | 推荐 |
| 5 | 识别失败模式 | "什么会出错?如何恢复?" | 推荐 |

### 知识类型分类

| 类型 | 特征 | Skill 放置位置 |
|-----|------|---------------|
| **程序性知识** | 步骤、流程、序列 | SKILL.md 主体 / rules/ |
| **声明性知识** | 事实、规则、约束 | reference/ |
| **启发式知识** | 专家判断、权衡 | rules/ 决策框架 |
| **参考性知识** | API 文档、规范 | reference/ |

### 核心元素提取清单

```
触发条件:
- 任务类型: "当分析..."、"当创建..."
- 文件类型: "处理 PDF..."、"Excel 数据..."
- 用户短语: "帮我..."、"我需要..."
- 领域关键词: [用户可能提到的专业术语]

核心工作流:
- 主流程 (快乐路径)
- 替代路径 (边界情况)
- 决策点 (判断分叉)
- 验证步骤 (确认成功)

失败模式:
- 常见错误
- 错误模式
- 恢复程序
- 预防策略
```

## 输出要求

1. **必须产出**：
   - 知识类型分类结果
   - 触发条件列表
   - 核心工作流描述
   - 决策点和失败模式 (如有)

2. **格式要求**：
   - 使用结构化列表
   - 标注知识类型
   - 明确优先级

3. **下一步引导**：
   - 完成后进入「架构设计」能力
   - 如信息不足，向用户请求补充

## 常见问题

**Q: 知识材料很长怎么办？**
A: 先提取目录/大纲，识别核心 vs 参考内容，核心内容详细分析，参考内容保留原文放 reference/

**Q: 知识类型混合怎么办？**
A: 按比例判断主要类型，混合型通常需要 rules/ + reference/ 结构

**Q: 触发条件不明确怎么办？**
A: 询问用户："这个 Skill 应该在什么情况下被使用？用户会说什么来触发它？"
