     1→# Tieban 模块使用说明（v8 对齐）
     2→
     3→本文档面向工程与产品使用，解释 Tieban 模块的 **输入/输出、核心流程、算法机制与可配置项**，并给出 API 用法与常见排错建议。本文与 `docs/tieban/tieban_spec_ultra_ultra_v8.md` 对齐。
     4→
     5→---
     6→
     7→## 1. 模块定位与输出形态
     8→
     9→Tieban 12k 的工程化流程为：
    10→
    11→```
    12→输入 → 标准化 → 起数 → 候选池 → 靠刻 → 锁定 → 命书输出
    13→```
    14→
    15→- **输入**：出生信息 + 事实（父母生肖、兄弟数等）
    16→- **输出（SSOT）**：结构化 JSON（命书四栏目 + 证据 + 追溯元数据）
    17→- **前端渲染**：从 JSON 显示条文原文与年龄区间
    18→
    19→---

    21→## 2. 使用入口
    22→
    23→### 2.1 API
    24→
    25→**`POST /api/tieban/init`**  
    26→一次性生成候选列表（不锁定）。
    27→
    28→**`POST /api/tieban/verify`**  
    29→追加或更新事实，重新筛选候选。
    30→
    31→**`POST /api/tieban/select`**  
    32→用户选择某候选进行锁定与出书。
    33→
    34→**`GET /api/tieban/report?run_id=...`**  
    35→获取结构化命书 JSON。
    36→
    37→**`POST /api/tieban/solve`**  
    38→一键流程：初始化后如果候选唯一则自动锁定并返回报告地址。
    39→

    42→## 3. 输入与标准化
    43→
    44→### 3.1 日期解释
    45→若输入为 `MM/DD/YYYY`，会统一解释为 `YYYY-MM-DD`（如 `02/11/1980 → 1980-02-11`）。
    46→
    47→### 3.2 真太阳时（TST）
    48→使用经度校正 + EOT（Equation of Time）计算真太阳时，并以此排四柱。
    49→
    50→### 3.3 边界上下文
    51→若出生时间接近节气或时辰边界，会生成多组 `pillars_contexts[]`，用于后续候选合并与比较。
    52→
    53→输出字段示例：
    54→```json
    55→{
    56→  "tst_time": "1980-02-11T14:04:03",
    57→  "boundary_flags": ["SOLAR_TERM_EDGE"],
    58→  "pillars_contexts": [
    59→    {"id": "A", "pillars": {"year":"庚申","month":"戊寅","day":"甲寅","hour":"辛未"}},
    60→    {"id": "B", "pillars": {"year":"庚申","month":"戊寅","day":"甲寅","hour":"辛未"}}
    61→  ]
    62→}
    63→```
    64→

    67→## 4. 核心算法机制
    68→
    69→### 4.1 Engine‑Stem（四柱天干取数）
    70→- 映射：`甲1 乙6 丙2 丁7 戊3 己8 庚4 辛9 壬5 癸0`
    71→- 顺序：**月 → 日 → 时 → 年**
    72→- 输出：四位数 base
    73→
    74→示例：`甲子年 丙寅月 壬申日 丙午时 → 2521`
    75→
    76→---
    77→
    78→### 4.2 Engine‑Hex（八卦加则密数表）
    79→- 以天干/地支配卦为上/下卦
    80→- 查密数表获得 2 个 base（primary + secondary）
    81→  - 若密数表缺失，可走公式（由 hex_constants）
    82→
    83→输出（每柱最多 2 个 base）：
    84→```
    85→year: [primary, secondary]
    86→month: [primary, secondary]
    87→day: [primary, secondary]
    88→hour: [primary, secondary]
    89→```
    90→
    91→---
    92→
    93→### 4.3 Engine‑DayMut（日主变卦）
    94→- 取日柱 **primary** base
    95→- 计算 `base ± 96 × k`，`k ∈ [-4..4]`
    96→- **默认包含 k=0（本值）**
    97→- secondary 仅在救援阶段启用
    98→
    99→---
   100→
   101→### 4.4 VerseIdMapping（条文序列）
   102→从 `base_ke = base + (ke - 1)` 生成等差序列：
   103→
   104→```
   105→range_min=1001, range_max=13000, step=96
   106→```
   107→
   108→应用规则：
   109→- `avoid_ten_rule`: 末位 0 过滤
   110→- `holes`: 显式跳过条文号
   111→- `mode`: `linear` 或 `wrap`
   112→  - `linear`：从 base_norm 递增到 range_max
   113→  - `wrap`：环形回绕至全周期
   114→
   115→---
   116→
   117→### 4.5 多事实靠刻语义
   118→对某事实 `fact`，若候选序列与 `match_verse_ids(fact)` 有交集，则命中。
   119→
   120→```
   121→intersection(candidate.sequence, match_verse_ids(fact)) ≠ ∅
   122→```
   123→
   124→**关键点**：不同事实允许命中不同条文，不要求同条文交集。
   125→
   126→---
   127→
   128→## 5. 候选池、筛选与救援
   129→
   130→### 5.1 候选池
   131→对每个上下文：
   132→1) Engine‑Stem/Hex/DayMut 生成 bases  
   133→2) 对每个 base 枚举 ke=1..8  
   134→3) 生成 verse_id_sequence
   135→
   136→### 5.2 全事实命中筛选
   137→若提交了 N 个事实，候选需对 **所有事实** 均有命中才保留。
   138→
   139→### 5.3 救援链路
   140→当候选为 0 时，依次尝试：
   141→1) `linear_offsets`  
   142→2) `day_mutation.include_secondary=true`  
   143→3) `secret_jump_search`  
   144→若仍为 0 → 返回 conflict。
   145→
   146→### 5.4 候选“已发生条文”预览（选盘辅助）
   147→为帮助用户在“未锁定”阶段对照过往经历，候选返回字段包含：
   148→- `occurred_preview`: 从该候选 **Grid 条文序列**中抽取“已发生/正在发生”的条文预览
   149→- `occurred_preview_as_of_age`: 计算预览时使用的虚岁（默认 `当前年份 - 出生年份 + 1`）
   150→
   151→筛选规则：
   152→- `age_start/age_end != 0/0`
   153→- `age_start <= occurred_preview_as_of_age`
   154→- 默认排除已用于 `evidence_preview` 展示的条文（避免重复）
   155→

   158→## 6. 锁定规则
   159→
   160→### 6.1 必填事实（可配置）
   161→锁定必填事实由 ruleset 配置决定：
   162→- 配置：`tieban_rulesets.rules.lock.required_fields`
   163→- 默认：`father_zodiac`、`mother_zodiac`（`siblings_count` 为可选事实）
   164→
   165→前端提示：若锁定时缺少必填事实，状态栏会提示用户补充缺失字段后再锁定。
   166→
   167→### 6.2 证据
   168→每个必填事实至少 1 条条文命中；证据写入 `evidence`。
   169→
   170→### 6.3 唯一性
   171→仅允许以下两种锁定：
   172→- **唯一候选**  
   173→- **唯一 true_ke**（仅锁 ke，base 不唯一）
   174→
   175→`lock_mode` 用于标注锁定类型：
   176→```
   177→unique_candidate | unique_ke
   178→```

   182→## 7. 命书输出结构
   183→
   184→### 7.1 四固定栏目
   185→1) 封面与卷首  
   186→2) 考刻定分  
   187→3) 先天六亲断语  
   188→4) 流年大运批导
   189→
   190→### 7.2 年表年龄来源
   191→年表严格使用 `tieban_verses.age_start/age_end`，`0/0` 不入年表。
   192→
   193→### 7.3 Generated vs Grid
   194→补充算法条文标记 `source="generated:..."`，主链路条文标记 `source="grid"`。  
   195→Generated 条文 **不得**用于锁定证据。
   196→

   199→## 8. 高级算法（补充引擎）
   200→
   201→### 8.1 太玄取数（Taixuan）
   202→将干支映射为太玄数，为前后卦与八卦滚法提供基础。
   203→
   204→### 8.2 前后卦取数（Front/Back）
   205→生成两条条文号，并归一化至 `1001..13000`。
   206→
   207→### 8.3 八卦滚法（Roll 48）
   208→8 卦 × 每卦 6 条，使用 **×100+ 组合公式**：
   209→
   210→```
   211→arrange = A[upper]*10 + A[lower]
   212→xian    = X[upper]*10 + X[lower]
   213→hou     = H[upper]*10 + H[lower]
   214→
   215→6 条：
   216→arrange*100 + xian
   217→arrange*100 + hou
   218→xian*100 + arrange
   219→xian*100 + hou
   220→hou*100 + arrange
   221→hou*100 + xian
   222→```
   223→
   224→### 8.4 冲突项配置
   225→见 `docs/tieban/tieban_v8_conflicts.md`：
   226→- `roll_policy.first_hex`：`opposite` / `mutual`
   227→- `verse_id_mapping.mode`：`linear` / `wrap`
   228→- `generated_sources`：`front_back` / `roll_48` / `fourmen`
   229→

   283→## 11. 常见错误与排查
   284→
   285→- `missing_required_facts`：缺少必填事实（服务端返回时可能附带缺失字段列表）  
   286→- `required_fact_not_matched`：事实未命中条文  
   287→- `lock_not_unique`：候选或 true_ke 未唯一  
   288→- `conflict=true`：救援链路后仍无法命中  
   289→
   290→排查顺序：
   291→1) 检查事实字段是否规范化（生肖枚举/数字字段）  
   292→2) 检查 rule set 是否加载正确  
   293→3) 对照 `tieban_v8_conflicts.md` 的配置项
