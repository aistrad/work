    63→## 与 v2.0 / v2.1 的差距（重点）
    64→
    65→以下为“规格中明确提到、但当前实现仍明显缺失/未闭环”的项目（优先级从高到低）：
    66→
    67→1. **工具调用闭环（Skill Tools in Agent Pipeline）**
    68→   - `AgentRuntime._invoke_tool()` 仍是空实现；八字/星座/MBTI 工具虽存在，但未接入对话流程
    69→2. **Vibe ID 站点与跨站 SSO 闭环**
    70→   - `apps/id` 为空；前端也缺少 `sso-callback` 等页面/跳转链路验证
    71→3. **Insight Cards 全链路**
    72→   - 后端有 `InsightGenerator` 与基础数据表，但前端未集成展示/保存/分享；触发规则与 UI 仍未联动
    73→4. **Proactive Reminder System 生产化**
    74→   - 有事件表/设置/通知服务与 API，但缺少定时调度（cron/队列）、push/wechat 等投递实现与权限收敛
    75→5. **Knowledge Layer 生产可用性**
    76→   - embedding/retrieval 依赖外部服务；向量写入/检索在 Postgres 的 `vector` 适配与索引/数据管道需要进一步验证
    77→6. **共享 UI 包与多站点生成**
    78→   - `packages/ui` 目录目前为空，尚未形成 “统一体验标准层” 的可复用组件库与站点模板引擎
    79→
    80→## 建议下一步（Phase 1 最小闭环）
    81→
    82→1. 在 `AgentRuntime._invoke_tool()` 里把 **bazi 排盘 + 基础分析** 接起来，形成 “输入出生信息 → 工具计算 → LLM 解读” 的闭环
    83→2. 把 guest landing 的出生信息输入改为 **真正调用后端工具**（即使不登录也能体验）
    84→3. 补齐 `apps/id`（或先用 `apps/web` 承担 SSO callback）做一次跨站 token 兑换端到端验证
    85→4. 将 reminder 初始化与每日扫描任务做成可部署的 cron（先落库 in-app 通知即可）
